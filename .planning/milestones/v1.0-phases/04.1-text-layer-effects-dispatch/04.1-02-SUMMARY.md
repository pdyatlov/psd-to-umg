## Walker Approach: Hand-Rolled vs PhotoshopAPI Descriptors::

The plan noted PhotoshopAPI's `Descriptors::DescriptorTaggedBlock` as a potential alternative. The hand-rolled walker was chosen because:
- It follows the same pattern already established for the lrFX flat binary decode in `ExtractLayerEffects`
- Avoids adding a dependency on `DescriptorStructure.h` which may have include-order conflicts with the PIMPL boundary
- The descriptor subset needed (Objc/bool/UntF/enum/VlLs skip) is small enough to implement cleanly

## lfx2 Discriminator: Empirical Finding

RESEARCH documented the discriminator as `m_Data[0..3] == 0x00000010`. The actual fixture bytes revealed:
- `m_Data[0..3]` = `0x00000000` (unknown flags field)
- `m_Data[4..7]` = `0x00000010` (descriptor version 16) -- the correct discriminator

The implementation checks `m_Data[4..7]` as the version marker. The secondary discriminator (looking for `FrFX` key after walking the descriptor) provides additional protection against false positives from other Unknown blocks.

## Deferred Work

- **Image-layer stroke rendering (D-12):** `FPsdLayerEffects.bHasStroke` is populated for ALL layer types (D-02) but rendering on image/group layers is not implemented. The data is dormant, ready for a future phase to consume.
- **Stroke position (inside/center/outside):** FrFX `Styl` enum is parsed by the walker but ignored. UE font outline is uniform; position mapping would require a future enhancement.
- **frameFXMulti VlLs format:** Newer Photoshop versions use `frameFXMulti` (VlLs) instead of `FrFX` (Objc) in the lfx2 descriptor. The walker handles `FrFX` only. Layers with `frameFXMulti` will have `bHasStroke = false` (safe fallback, no crash). Future phase can extend the walker to check `frameFXMulti[0]` as well.
- **text_complex bHasComplexEffects:** All TextEffects.psd layers have lrFX blocks with `isdw`/`bevl` records (always present in Photoshop-written lrFX, regardless of enabled state). The generator will fire the D-11 warning for all of them. A future improvement would be to check the `enabled` byte in isdw/bevl records before setting `bHasComplexEffects`.

## Post-Ship Hotfix 2026-04-15

**Problem confirmed:** PhotoshopAPI v0.9 silently drops `lfx2` tagged blocks — they never appear in `unparsed_tagged_blocks()`. Diagnostic logging added after ship showed 0 Unknown blocks per layer despite `lfx2` existing 4 times in `TextEffects.psd` (empirically: raw offsets 24696, 40414, 56124, 71886). The `ExtractLfx2Stroke` walker loop over `unparsed_tagged_blocks()` was dead code.

**Fix (commit f728c83):**

1. Removed all `[DIAG]` diagnostic logging from `ExtractLfx2Stroke`.
2. Extracted the FrFX descriptor walker body into `ParseFrFXDescriptor(TArrayView<const uint8>, FPsdStrokeInfo&)` — a pure function over raw bytes.
3. Added `ScanRawLfx2Blocks`: scans raw PSD file bytes for `8BIMlfx2` signatures, scans backwards per-hit for the nearest `8BIMluni` block to recover the layer name (UTF-16BE), calls `ParseFrFXDescriptor` per hit, and builds a `TMap<FString, FPsdStrokeInfo>`.
4. Added `ParseLuniLayerName`: parses the luni block's UTF-16BE char sequence into an `FString`.
5. `ParseFile` now loads raw bytes via `FFileHelper::LoadFileToArray` before the PSAPI layer walk and passes the map to `ConvertLayerRecursive`.
6. `ExtractLfx2Stroke` signature changed to `(const FString& LayerName, const TMap<FString,FPsdStrokeInfo>&, FPsdLayer&)` — does a map lookup, no PhotoshopAPI types involved.

Files modified: `Source/PSD2UMG/Private/Parser/PsdParser.cpp` only. PIMPL boundary, `FPsdLayerEffects` shape, `RouteTextEffects`, mapper, and spec files untouched.

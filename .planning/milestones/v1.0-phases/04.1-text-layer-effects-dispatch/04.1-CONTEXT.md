# Phase 4.1: Text Layer Effects Dispatch (Gap Closure) — Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the TEXT-03 / TEXT-04 dispatch gap surfaced by the v1.0 milestone audit. Photoshop Layer-Style **Stroke** and **Drop Shadow** applied to text layers currently fall through the Phase 5 generator paths (`bHasComplexEffects` flatten-with-no-pixels warning, or image-only drop-shadow gate) and are silently dropped. Phase 4.1 routes text-parented stroke and drop-shadow effects into `FPsdTextRun` so `FTextLayerMapper` applies them via `FSlateFontInfo::OutlineSettings`, `UTextBlock::SetShadowOffset`, and `UTextBlock::SetShadowColorAndOpacity`.

**In scope:**
- Add first-class Layer-Style Stroke (`oglw`) extraction to `ExtractLayerEffects` in `PsdParser.cpp`
- Extend `FPsdLayerEffects` with `bHasStroke / StrokeColor / StrokeSize`
- Un-comment and populate `FPsdTextRun::ShadowOffset` and `FPsdTextRun::ShadowColor`
- Parser post-process: when `OutLayer.Type == EPsdLayerType::Text`, route `Effects.DropShadow*` → `Text.Shadow*` and `Effects.Stroke*` → `Text.Outline*`, then clear the corresponding `bHas*` flags on `Effects` so downstream generator paths don't double-render
- Precedence: Layer-Style stroke overrides any TySh character-level stroke already populated by Phase 4
- `FTextLayerMapper` reads the new shadow fields and calls `SetShadowOffset` + `SetShadowColorAndOpacity`
- DPI conversion (×0.75) applied to both stroke size and shadow offset X/Y (same convention as TEXT-01)
- Fixture PSD + automation spec coverage proving the round-trip

**Out of scope:**
- Rendering stroke on non-text (image) layers — parser extracts the data but this phase does not render it; v2/future phase consumes
- Text-material approximation for gradient / pattern / bevel on text layers — keep today's warning-only no-op
- Inner shadow, gradient overlay, pattern overlay, bevel on text — remain classified `bHasComplexEffects`; text hits the existing warning path
- Per-character (multi-run) shadow / stroke — still covered by Phase 4's "first run flattens" pattern

</domain>

<decisions>
## Implementation Decisions

### Stroke Extraction (TEXT-03)
- **D-01:** Add first-class Layer-Style Stroke extraction to `ExtractLayerEffects`. New fields on `FPsdLayerEffects`:
  ```cpp
  bool bHasStroke = false;
  FLinearColor StrokeColor = FLinearColor::Transparent;
  float StrokeSize = 0.f;   // PSD pixels
  ```
  **CORRECTION (2026-04-15, per 04.1-RESEARCH.md):** `oglw` = **Outer Glow**, NOT Stroke. Layer-Style Stroke lives in the `lfx2` (object-based effects) tagged block under descriptor key `FrFX` (FrameFX), not in the `lrFX` loop. PhotoshopAPI v0.9 does not register `lfx2` — it surfaces as `Unknown` in `unparsed_tagged_blocks()` and is identified by the leading 4-byte big-endian `0x00000010` descriptor version marker. Decoding `FrFX` requires walking Photoshop's recursive descriptor format (ostype dispatch for `Objc` / `bool` / `doub` / `UntF`) — approximately 50-80 lines. Color sub-descriptor uses self-labeled `Rd  ` / `Grn ` / `Bl  ` doubles on a 0-255 scale (no ARGB ambiguity — the §7a audit concern does not apply to the lfx2 path). Size is a `doub` in PSD pixels. See 04.1-RESEARCH.md §Stroke Decoding for the full walker spec.
- **D-02:** Stroke extraction runs for **all layer types**, not just text. Non-text consumers are out of scope for this phase but data is available for future use (image-layer stroke, smart-object stroke).

### Stroke Precedence (Text)
- **D-03:** When a text layer has BOTH TySh character-level stroke (Phase 4's `Text.OutlineSize/Color` populated from `style_run_stroke_flag`) AND Layer-Style Stroke, **Layer-Style wins**. The parser post-process **overwrites** `Text.OutlineColor/Size` with the Layer-Style values when `Effects.bHasStroke == true`.
- **D-04:** No warning required on precedence conflict — designers routinely use both; overwriting silently matches designer intent ("stroke the whole text block").

### Effects → Text Routing (parser post-process)
- **D-05:** Routing location: **parser** (not generator, not mapper). After `ExtractLayerEffects` and the type-resolution pass, if `OutLayer.Type == EPsdLayerType::Text`:
  1. If `Effects.bHasDropShadow`: `Text.ShadowOffset = Effects.DropShadowOffset; Text.ShadowColor = Effects.DropShadowColor;` then clear `Effects.bHasDropShadow` so the generator's image-shadow path (line ~212 of `FWidgetBlueprintGenerator.cpp`) does not fire a redundant sibling-UImage shadow.
  2. If `Effects.bHasStroke`: `Text.OutlineColor = Effects.StrokeColor; Text.OutlineSize = Effects.StrokeSize;` then clear `Effects.bHasStroke`.
  3. Leave `Effects.bHasColorOverlay` and `Effects.bHasComplexEffects` untouched — those keep existing semantics.
- **D-06:** Routing is a dedicated helper (`RouteTextEffects(FPsdLayer&)` or similar) called from `ConvertLayerRecursive` after both text and effects data are populated. Centralized so generator/mapper code stays unaware of it.

### Text Shadow Fields (TEXT-04)
- **D-07:** Un-comment and enable the previously-deferred fields in `FPsdTextRun`:
  ```cpp
  FVector2D ShadowOffset = FVector2D::ZeroVector;  // UMG pixels (post-DPI)
  FLinearColor ShadowColor = FLinearColor::Transparent;
  ```
  The Phase 4 comment block (`// TEXT-04 fields intentionally omitted ...`) is deleted; this phase IS the referenced "4.1 gap closure".
- **D-08:** `FTextLayerMapper` applies the fields after the existing outline block:
  ```cpp
  if (!Layer.Text.ShadowOffset.IsZero() || Layer.Text.ShadowColor.A > 0.f)
  {
      TextWidget->SetShadowOffset(Layer.Text.ShadowOffset);
      TextWidget->SetShadowColorAndOpacity(Layer.Text.ShadowColor);
  }
  ```
  The deferred-delivery comment at `FTextLayerMapper.cpp:106-112` is removed.

### DPI Conversion
- **D-09:** Apply ×0.75 (TEXT-01 convention) to **both** stroke size and shadow offset X/Y. Conversion happens once, at the same site:
  - Option A (preferred by symmetry): in the parser post-process when copying `Effects.*` → `Text.*`, so `FPsdTextRun` carries UMG-space values and the mapper stays a pass-through. Document field units as UMG pixels in the struct comment.
  - Option B: in the mapper (current outline pattern at `FTextLayerMapper.cpp:97`). Choose whichever the planner deems cleaner; both are acceptable provided it happens exactly once. Default recommendation: Option A for consistency with how `Text.SizePx` already works (mapper converts on the fly — so actually Option B matches existing pattern). **Planner chooses; document the choice in the plan.**
- **D-10:** Stroke extraction stores raw PSD pixels in `Effects.StrokeSize`. Shadow offset in `Effects.DropShadowOffset` is already raw PSD pixels (Phase 5 did no conversion). Both get ×0.75 at the chosen conversion site.

### Complex Effects on Text
- **D-11:** Text layer with gradient overlay / pattern overlay / inner shadow / bevel (flagged as `bHasComplexEffects`) continues to emit the existing `UE_LOG(LogPSD2UMG, Warning, "Layer '%s' has complex effects but no pixel data for flatten — effects ignored")` at `FWidgetBlueprintGenerator.cpp:77`. No upgrade to dialog diagnostics, no approximation attempt. Zero scope creep; gap-closure phase stays narrow.

### Scope Boundary (Non-Text Stroke)
- **D-12:** Parser extracts stroke for all layer types (D-02). This phase renders stroke on **text layers only**. Image layers with Layer-Style stroke: `Effects.bHasStroke` is populated but the generator does not consume it — the data sits dormant for a future phase to render (e.g., via UBorder, material, or sibling UImage outline). Document this as a known deferral in `04.1-VERIFICATION.md`.

### Double-Render Guard
- **D-13:** The parser post-process clears `Effects.bHasDropShadow` on text layers (per D-05). This is the authoritative guard against the image-shadow sibling path double-rendering. No additional `LayerPtr->Type == EPsdLayerType::Text` check is added at the generator site — keep the generator code shape intact.

### Fixture & Automation
- **D-14:** Add a fixture PSD (or extend an existing one) under `Source/PSD2UMG/Tests/Fixtures/` that contains at minimum: one text layer with Layer-Style Stroke, one text layer with Drop Shadow, one text layer with both, and one text layer with a complex effect (e.g., gradient overlay) to prove the warning-only path still fires. Planner decides filename (e.g., `TextEffects.psd`) and whether to extend `Typography.psd`.
- **D-15:** Automation spec coverage: extend `FTextLayerMapperSpec` (or add `FTextEffectsSpec`) to assert `TextWidget->GetFont().OutlineSettings.OutlineSize` matches expected post-DPI value, `GetShadowOffset()` and `GetShadowColorAndOpacity()` match, and that a non-shadow text layer still renders with zero offset / transparent color.

### Claude's Discretion
- DPI conversion site (parser post-process vs mapper) — D-09 Option A vs B; planner chooses and documents
- Helper name for the routing function (`RouteTextEffects`, `PostProcessTextLayer`, etc.)
- Whether to add a `bHasStroke`-clear assertion to the mapper (defensive) or rely purely on parser post-process
- Exact fixture file name and whether to share with Phase 4's `Typography.psd`
- Whether the planner wants to gate Layer-Style-overrides-TySh behind a log line (D-04 says silent by default, but a `LogPSD2UMG Verbose` line is fine)

</decisions>

## Existing Code Insights

### Reusable Assets
- `ExtractLayerEffects` in `PsdParser.cpp` — already handles the lrFX framing (version/effect-count loop, per-effect `EffectEnd` advance). Adding `oglw` is a new `else if` branch inside the existing loop.
- `FPsdTextRun::OutlineColor/OutlineSize` — already populated by TySh path; the Layer-Style path simply overwrites them.
- `FTextLayerMapper` outline block — working template for the new shadow block immediately below it.
- `FPsdParseDiagnostics` — Phase 2 warning channel; reuse for any new diagnostics (default: none; D-11 keeps the existing generator warning).

### Established Patterns
- PIMPL boundary: all PhotoshopAPI calls stay inside `Source/PSD2UMG/Private/Parser/`. Public headers never see PhotoshopAPI types.
- Best-effort effect data: zero-defaults + `bHas*` flag pattern (Phase 4 outline + Phase 5 effects). New stroke fields follow the same shape.
- DPI conversion at the mapper (TEXT-01 baseline, TEXT-03 outline). D-09 lets the planner pick parser vs mapper for this phase.
- Warning-only for unsupported effects (D-08 Phase 3 skip-and-warn; `FWidgetBlueprintGenerator.cpp:77`). Phase 4.1 preserves this for text+complex.

### Integration Points
- `FPsdLayerEffects` struct — additive extension (3 new fields). No existing consumer iterates the struct; all read specific named fields.
- `ConvertLayerRecursive` post-step — new call to route text effects. Called once per layer, after effects and text extraction both run.
- `FTextLayerMapper::Map` — new shadow apply block (3-5 lines after the outline block).
- No changes needed to `FWidgetBlueprintGenerator` core dispatch — the parser post-process clears the shadow flag so the image-shadow path stays image-only automatically.

<specifics>
## Specific Ideas

- The `oglw` effect sub-record shape is the primary research question for the planner. If PhotoshopAPI exposes a higher-level accessor than `unparsed_tagged_blocks()` for stroke, prefer that; otherwise decode manually like `dsdw` does.
- Layer-Style stroke position (inside / center / outside) is not representable on `FFontOutlineSettings` — just use outside-equivalent (UE outline is uniform). Document as a v2 limitation if needed.
- Shadow opacity encoding matches `dsdw` (Phase 5): opacity byte / 255 → alpha. Color RGB channels in the same ARGB-ish order as sofi/dsdw (Phase 2 empirical finding, still pending TEXT-03 host-UE confirmation per audit §7a — this phase can piggyback that check).
- When Layer-Style stroke overwrites TySh stroke (D-03), keep a `LogPSD2UMG Verbose` trace for debugging ("Layer-Style stroke overriding TySh stroke on layer '%s'"). Not required, but cheap.

</specifics>

- **Stroke position (inside/center/outside)** — UE font outline is uniform; Photoshop position attribute ignored.
- **Preview-dialog surfacing of "effects ignored" warning** — D-11 keeps Output-Log-only; upgrade candidate for the Phase 7 Editor UI polish phase.

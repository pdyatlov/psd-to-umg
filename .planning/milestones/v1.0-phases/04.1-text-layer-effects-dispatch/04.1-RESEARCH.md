## Summary

The Phase 4.1 gap has two halves with very different complexity profiles:

**TEXT-04 (Drop Shadow on text) is straightforward.** The `dsdw` effect is already decoded from the `lrFX` block into `FPsdLayerEffects::bHasDropShadow`, `DropShadowColor`, and `DropShadowOffset` by the Phase 5 `ExtractLayerEffects` function. The only work is (1) un-commenting `FPsdTextRun::ShadowOffset/ShadowColor`, (2) adding a 5-line routing helper in `ConvertLayerRecursive` that copies these fields to the text payload and clears the Effects flag, and (3) adding a 3-5 line shadow-apply block in `FTextLayerMapper::Map` after the existing outline block. UE API: `UTextBlock::SetShadowOffset(FVector2D)` and `UTextBlock::SetShadowColorAndOpacity(FLinearColor)` take UMG pixel units — same coordinate system as `SetShadowOffset`. No surprises.

**TEXT-03 (Stroke on text) is the primary unknown that CONTEXT.md has a factual error about.** CONTEXT.md D-01 states the stroke effect uses key `"oglw"` inside the `lrFX` loop. This is incorrect. Per direct inspection of the psd-tools reference implementation (`EffectOSType` enum) and the Adobe spec, `"oglw"` is **Outer Glow**. The complete `lrFX` block contains only these effect keys: `cmnS` (common state), `dsdw` (drop shadow), `isdw` (inner shadow), `oglw` (outer glow), `iglw` (inner glow), `bevl` (bevel), `sofi` (solid fill). **Stroke is absent from `lrFX`.**

Layer-Style Stroke in Photoshop CS and later is stored in the `lfx2` (object-based effects) tagged block, using the descriptor key `FrFX` (`FrameFX`). PhotoshopAPI v0.9 does not register `"lfx2"` as a known `TaggedBlockKey` — it maps to `Unknown` — but raw bytes are still accessible via `unparsed_tagged_blocks()`. Decoding requires walking Photoshop's descriptor binary format, which is more involved than the flat `lrFX` structure.

**Primary recommendation:** For TEXT-03, decode stroke from the `lfx2` descriptor block (manually or via PhotoshopAPI's `Descriptors::DescriptorBase` infrastructure which is present in the vendored headers). For TEXT-04, the routing is purely a wiring task on already-decoded data.

## Architecture Patterns

### CRITICAL CORRECTION: `oglw` ≠ Stroke

The CONTEXT.md decision D-01 states: *"Parser decodes the `oglw` effect key alongside `sofi` / `dsdw` / `isdw` / `bevl`."*

**This is incorrect.** `oglw` is Outer Glow. The `lrFX` block has no stroke entry. Confirmed by:
1. Direct inspection of psd-tools `constants.py` `EffectOSType` enum (values: `cmnS`, `dsdw`, `isdw`, `oglw`, `iglw`, `bevl`, `sofi`)
2. psd-tools `effects_layer.py` `EffectsLayer.EFFECT_TYPES` dict — no stroke entry
3. The existing PsdParser.cpp comment at line 320: *"Photoshop Layer Style strokes (fx → Stroke) are stored in the lfx2 descriptor which PhotoshopAPI v0.9 does not expose."*

**Stroke location in PSD files:** `lfx2` tagged block (Photoshop 6.0+), descriptor key `FrFX` (`FrameFX`). This block uses Photoshop's descriptor binary format (not the flat fixed-field format of `lrFX`).

### Pattern 2: lfx2 Descriptor Decode (new, for TEXT-03 stroke)

The `lfx2` block is NOT in `PhotoshopAPI::Enum::TaggedBlockKey`; it surfaces as `Unknown` key in `unparsed_tagged_blocks()`. The block signature is `"8BIM"` + `"lfx2"` + uint32 length + descriptor bytes.

**Access path:**
```cpp
for (const auto& Block : InLayer->unparsed_tagged_blocks())
{
    // lrFX = fxLayer enum; lfx2 = Unknown key, raw 4-char key in Block->m_Key region
    // Must compare raw key bytes, not the enum, because lfx2 is not in the map.
}
```

**However**: PhotoshopAPI's `TaggedBlock::getKey()` returns the enum, and `Unknown` covers all unregistered keys. The raw 4-char key is stored as the block's signature bytes. We need to check `Block->m_Data` or look at the block type:

```cpp
// The tagged block stores the raw 4-char key — access via Block->getKey() won't
// distinguish lfx2 from other Unknown blocks. Need to iterate and check the raw
// key stored in the block's Signature field or skip offset within m_Data.
```

**Implementation option for the planner:** Since PhotoshopAPI doesn't expose the raw key string on `TaggedBlock`, the planner has two sub-options:
- **Option A (simpler):** Scan all blocks for `TaggedBlockKey::Unknown` and look for `"lfx2"` at a fixed byte offset within `Block->m_Data` (the block data includes version + descriptor starting after the 4-char key+length header that PhotoshopAPI strips).
- **Option B (correct):** PhotoshopAPI's `TaggedBlock` struct has a `m_Signature` field (accessible). After confirming `getKey() == Unknown`, compare raw data prefix bytes against `"lfx2"`.

Actually — checking the `TaggedBlock.h` header: the struct has `m_Signature` (Signature type) and `m_Key` (private, accessed via `getKey()`). The raw 4-char string is the **map key** in `taggedBlockMap`. Since `"lfx2"` is not in the map, any block with that 4-char ID will report `Unknown`. The 4-char ID is embedded at offset 4 of the block's on-disk location, **not** in `m_Data` (m_Data holds only the payload after the header). So the planner cannot distinguish `lfx2` from other Unknown blocks via `m_Data` alone without a workaround.

**Recommended workaround:** Verify the raw key using `Block->m_Signature` + cross-check the first 4 bytes of `m_Data` against the descriptor version marker `\x00\x00\x00\x10` (uint32 = 16, the standard descriptor version), OR — more robustly — add a custom string comparison on the Block pointer's underlying key. Since `TaggedBlock` exposes `m_Data` as `std::vector<std::byte>`, the block data for `lfx2` starts with: `uint32 descriptor_version` (= 16, 4 bytes) then the full Photoshop descriptor. The distinctive fingerprint vs. other Unknown blocks is the leading `0x00000010` descriptor version.

**lfx2 descriptor structure (partial, for stroke):**
```
[uint32] descriptor version = 16
[descriptor] {
  "null" class_id
  key-value pairs:
    "FrFX" → Descriptor {
      "enab" → bool (enabled)
      "Styl" → enum (position: inside/center/outside)
      "PntT" → enum (fill type: solid/gradient/pattern)
      "Sz  " → UntF (size in pixels)
      "Clr " → Descriptor (RGBA color)
      "Opct" → UntF (opacity percent)
      "overprint" → bool
    }
}
```

Key fields for this phase: `"enab"` (enabled bool), `"Sz  "` (size as UnitFloat, pixels), `"Clr "` (color descriptor with RGBA), `"Opct"` (opacity 0-100 percent).

**Simpler alternative:** Since PhotoshopAPI's `Descriptors::` namespace has full descriptor read/write infrastructure (see `DescriptorStructure.h` — `DescriptorBase`, `DescriptorTaggedBlock`, etc.), the planner may be able to construct a `Descriptors::DescriptorTaggedBlock` and call its `read()` method on a `File` created from the raw bytes. This avoids hand-rolling a descriptor walker. **Confidence: MEDIUM** — this depends on whether the PhotoshopAPI descriptor types are accessible from the PIMPL boundary inside `PsdParser.cpp`.

### Anti-Patterns to Avoid
- **Adding `"oglw"` branch to the `lrFX` loop:** `oglw` is Outer Glow, not Stroke. This would parse the outer glow record (blur, intensity, color, blend mode) as if it were a stroke, producing incorrect data.
- **Assuming `lfx2` follows the same flat binary format as `lrFX`:** The `lfx2` block uses Photoshop's descriptor binary format (recursive key-value tree), not fixed-field offsets.
- **Double-converting DPI:** The existing `dsdw` decode in `ExtractLayerEffects` stores raw PSD pixels in `DropShadowOffset`. Applying ×0.75 in both `RouteTextEffects` AND `FTextLayerMapper` would double-scale. Apply once at the mapper (Option B) to match the existing `OutlineSize * 0.75f` pattern.

## Common Pitfalls

### Pitfall 1: `oglw` is Outer Glow, not Stroke
**What goes wrong:** Adding `else if (FCStringAnsi::Strcmp(Key, "oglw") == 0)` to the `lrFX` loop and reading blur/intensity/color fields as if they were stroke size/color. The decode partially succeeds (both effect types have a color sub-record), producing a non-zero `StrokeSize` derived from the outer-glow blur radius, which then draws an incorrect outline on text.
**Why it happens:** CONTEXT.md D-01 incorrectly identifies `oglw` as the stroke key.
**How to avoid:** Do not add `oglw` to `ExtractLayerEffects` for stroke. Stroke is in `lfx2`, not `lrFX`.
**Warning signs:** `bHasStroke` fires on layers that have outer glow but no stroke in Photoshop's Layer Style dialog.

### Pitfall 2: lfx2 Identification Among Unknown Blocks
**What goes wrong:** `unparsed_tagged_blocks()` returns all blocks including `lfx2`, but since `lfx2` is not in PhotoshopAPI's `taggedBlockMap`, `getKey()` returns `Unknown`. Iterating for `TaggedBlockKey::fxLayer` misses it entirely; iterating for `Unknown` finds it but also finds other unknown blocks.
**Why it happens:** PhotoshopAPI v0.9 doesn't register `lfx2`.
**How to avoid:** After confirming `getKey() == Unknown`, check block data prefix for descriptor version marker (`m_Data[0..3]` = `0x00000010` = descriptor version 16). Alternative: check if the block's raw key (from the PSD stream offset embedded in `m_Offset`) matches `"lfx2"` via `Block->m_Signature`.

### Pitfall 3: DPI Double-Conversion
**What goes wrong:** Applying ×0.75 in `RouteTextEffects` (Option A) and again in `FTextLayerMapper` by analogy with the existing `OutlineSize * 0.75f` line.
**Why it happens:** D-09 allows either site, and the mapper's `OutlineSize * 0.75f` is a natural copy target.
**How to avoid:** Choose one site. **Recommendation: Option B (mapper)** because: (a) the existing outline block already does `OutlineSize * 0.75f` — a reviewer expects the same pattern for shadow; (b) `FPsdTextRun` can then document all its fields as "raw PSD pixels" uniformly; (c) the mapper is already the DPI conversion layer per TEXT-01.

### Pitfall 4: Shadow Fields Not Cleared — Double Render
**What goes wrong:** `Effects.bHasDropShadow` stays `true` on a text layer after routing. The generator's image-shadow path at `FWidgetBlueprintGenerator.cpp:212` checks `LayerPtr->Effects.bHasDropShadow && LayerPtr->Type == EPsdLayerType::Image` — the `Image` type-guard already prevents firing. BUT the Phase 5 `bHasDropShadow` log line at ~line 499 would still trigger, causing confusing log output.
**Why it happens:** Missing the clear step in `RouteTextEffects`.
**How to avoid:** Always clear `Effects.bHasDropShadow = false` in `RouteTextEffects` for text layers (D-13). The type check at the generator image-shadow path is a secondary safety net, not the primary guard.

### Pitfall 5: Test Spec Needs Updating — `text_stroked` Assertion Reversal
**What goes wrong:** `PsdParserSpec.cpp:266-275` currently asserts `L->Effects.bHasComplexEffects == true` for the `text_stroked` layer (because Layer-Style stroke was unhandled and fell through to `bHasComplexEffects`). After Phase 4.1, that same layer should have `bHasStroke == true` and `bHasComplexEffects == false`.
**Why it happens:** The old test was correct for the old behavior.
**How to avoid:** Update `PsdParserSpec.cpp` test `"text_stroked has complex effects from Layer Style Stroke"` to assert the new path: `bHasStroke == true`, `Effects.bHasComplexEffects == false`, `Text.OutlineSize > 0`.

for (const auto& Block : InLayer->unparsed_tagged_blocks())
{

### DPI Conversion Golden Values
For a shadow with Distance=10px, Angle=135°:
- Raw offset: X = 10 × cos(135°) ≈ -7.07, Y = 10 × sin(135°) ≈ 7.07 PSD pixels
- UMG pixels (×0.75): X ≈ -5.30, Y ≈ 5.30

For stroke Size=4px:
- UMG pixels: 4 × 0.75 = 3.0 → `RoundToInt` → 3

These should be the expected values in spec assertions (using `TestTrue(TEXT(...), FMath::IsNearlyEqual(shadow.X, -5.3f, 0.1f))`).

## Open Questions

1. **lfx2 block identification without raw key access**
   - What we know: `TaggedBlock::getKey()` returns `Unknown` for `lfx2`; `m_Data` holds payload bytes only.
   - What's unclear: Whether the leading `0x00000010` descriptor version prefix is a reliable discriminator vs. other Unknown blocks that also start with a descriptor.
   - Recommendation: Treat as HIGH confidence for modern PSDs (all descriptor-based tagged blocks use version 16); add a try/catch around the descriptor walk so malformed data is silently skipped.

2. **lfx2 Descriptor Parsing — Use PhotoshopAPI or hand-roll?**
   - What we know: `Descriptors::DescriptorBase` in `DescriptorStructure.h` is present in vendored headers but requires a `File` object wrapping the raw bytes.
   - What's unclear: Whether constructing a `File` from `std::vector<std::byte>` is feasible (PhotoshopAPI's `File` abstraction may require a disk path). If not, hand-rolling the descriptor walk is necessary.

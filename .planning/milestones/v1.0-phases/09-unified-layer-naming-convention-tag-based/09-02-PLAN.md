<task type="checkpoint:decision" gate="blocking">
  <name>Task 1: Decide reimport identity-key strategy (Risk #2)</name>
  <decision>
    After this phase, a widget's `FName` is `ParsedTags.CleanName` (tags stripped). But the Phase 7 reimport system keys widget identity on the verbatim PSD `Layer.Name`. Adding or removing a single tag in Photoshop would orphan every widget under old reimport matching. How do we handle this?
  </decision>
  <context>
    - Existing WBPs imported in Phase 3-8 have widget FNames built from the pre-tag layer names (e.g., `Button_Start`). After Plan 03 retags fixture PSDs, a fresh import would produce widget FName `Start`. Existing user WBPs cannot be auto-migrated.
    - CONTEXT.md D-15 mandates NO backwards compatibility.
    - Reference: 09-RESEARCH.md Risk #2 + Pitfall 4; Phase 7 D-05 reimport identity contract.
  </context>
  <options>
    <option id="option-a">
      <name>Match by CleanName, accept the break</name>
      <pros>
        - Simplest — no extra metadata.
        - Consistent with D-15 hard cutover philosophy.
        - Reimport behavior stable once user is fully on tag syntax (adding/removing a tag only breaks identity if it changes CleanName, which is rare).
      </pros>
      <cons>
        - Adding `@anchor:tl` is fine (CleanName unchanged), but changing the widget name portion breaks identity (same as today).
        - Existing pre-Phase-9 WBPs become fully orphaned on first reimport — designer must delete + reimport.
        - Migration guide must explicitly state this.
      </cons>
    </option>
    <option id="option-b">
      <name>Store original Layer.Name as widget metadata, match by it</name>
      <pros>
        - Preserves Phase 7 reimport contract verbatim.
        - Designer can freely toggle tags without orphaning widgets.
      </pros>
      <cons>
        - Requires adding an annotation slot or FString property to WidgetTree storage — non-trivial UE5 plumbing.
        - Contradicts D-15 spirit slightly (keeps a legacy-ish identity path).
        - More code to test.
      </cons>
    </option>
  </options>
  <resume-signal>Select: option-a or option-b</resume-signal>
</task>
